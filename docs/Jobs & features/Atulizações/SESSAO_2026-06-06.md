# Sessao 2026-06-06 - ChatDelivery local persistente para Memory Cards

## Contexto

A sessao partiu de um problema reproduzivel no runtime de Memory Cards: o card era acionado, mas a entrega de contexto dependia demais da leitura da vault/Supabase e podia voltar vazia. O resultado era um card aparentemente valido na interface, mas sem texto util na etapa de entrega para o chat.

O objetivo desta rodada foi reduzir essa dependencia, mantendo o escopo limitado a entrega de contexto e persistencia local de memoria, sem mexer em schema, RLS, chunks, Writer, prompt ou CRUD amplo.

---

## 1. Problema observado

Nos logs, o fluxo de entrega do card aparecia com comportamento inconsistente:

- `ChatDelivery` era chamado ao iniciar o chat do card;
- a leitura remota retornava `textLength: 0` em alguns casos;
- o card continuava visivel na UI, mas nao tinha uma entrega confiavel de contexto para a conversa;
- as regras/conhecimento do card precisavam sobreviver mesmo quando a API ficasse vazia ou falhasse.

Isso indicou a necessidade de uma memoria persistente fora da Supabase, no proprio desktop.

---

## 2. Solucao aplicada

### 2.1 Entregador local persistente

**Arquivo criado:** `js/Chatdelivery.js`

Foi criado um entregador de contexto para Memory Cards com prioridade de leitura assim:

1. cache local persistente em disco;
2. leitura da vault quando disponivel;
3. fallback para o proprio conteudo canônico do card carregado em memoria;
4. persistencia local do texto entregue para reutilizacao futura.

### 2.2 Cache fora da Supabase

O `ChatDelivery` passou a suportar:

- leitura local do card entregue anteriormente;
- gravacao local do texto consolidado;
- reutilizacao do cache em chamadas seguintes;
- logs curtos para auditoria sem expor conteudo grande.

O cache foi pensado para reduzir a dependencia da API em cenarios onde a vault devolve vazio ou nao consegue reconstruir o card a tempo.

### 2.3 Salvamento offline-first das regras

**Arquivo ajustado:** `js/memory-cards-runtime.js`

O salvamento de instrucoes/regras do Memory Card passou a:

- atualizar o catalogo em memoria;
- gravar a copia local persistente;
- tentar o endpoint remoto depois;
- manter o cache local mesmo se a API falhar.

Isso impede perda do texto canônico do card quando o backend nao responde como esperado.

### 2.4 Documentacao de arquitetura

**Arquivo ajustado:** `docs/architecture.json`

A documentacao foi atualizada para refletir que o `ChatDelivery` agora funciona com cache local persistente e nao depende apenas da vault remota.

---

- `js/Chatdelivery.js`

## 4. Comportamento final

Depois do patch:

- um Memory Card pode ser entregue mesmo se a leitura remota vier vazia;
- o primeiro texto util monta e grava cache local;
- a segunda abertura reaproveita o cache sem precisar reconstruir tudo pela API;
- o salvamento de regras/conhecimento nao se perde se a API falhar;
- o fluxo continua sem alterar schema ou RLS.
